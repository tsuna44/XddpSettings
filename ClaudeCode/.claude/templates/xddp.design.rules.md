# XDDP Step 06 設計品質ルール（CHD: 変更設計書）

> このファイルは `xddp.06.design` スキルが参照する設計品質ゲート定義。
> ルールのみを変更する場合はこのファイルだけを編集する。

---

## R1. 詳細設計（第3章）の記述ルール

第3章は **仕様（SP）単位** で記述すること。ファイル単位で章を分けてはならない。

### 構成ルール
- 各サブセクションの見出しは「SP-{番号}：{仕様名}」とする
- CRS の SP を順番に走査し、設計内容のある SP をすべて記述する
- 1つの SP が複数ファイルにまたがる場合は、そのセクション内でファイルごとにBefore/Afterを記述する
- 変更のないファイルはこの章に記載しない
- ファイル単位の全体像はセクション2「変更対象ファイル一覧」が担う

### Before/After の記述方針
- **Before：** 該当箇所のみを抜粋する（ファイル全体を貼らない）
- **After：** 変更後の該当箇所を記述する
- 新規追加の場合は Before を「（新規追加）」と記載する
- 変更箇所の説明は箇条書きで「何をどう変えるか」を明記する

### トレーサビリティマトリクス（第4章）
全SP × 変更ファイル × 変更関数の対応表を必ず作成する。  
第3章の記述と整合していること。

## R2. 図の作成ルール（Mermaid）

**基本方針：変更前の構造は SPO図で示し、変更後のアプローチは DSN図で示す。両者を役割分担して CHD に含めること。**

### SPO図・DSN図の役割分担（優先）

該当する SP セクションに以下を含めること：

- **変更前の構造（SPO図を転載）**: SPO モジュール個別（`modules/*-spo.md`）またはクロスモジュール（`cross-module/`）の
  状態遷移図・クラス図・シーケンス図・構造図等から、変更箇所・影響箇所を示す図を転載する。
  キャプションに「変更前 / 出典: SPO-{CR}（{モジュール名またはクロスモジュール}）」と記載する。
- **変更後のアプローチ（DSN図を転載）**: DSN（`DSN-{CR}.md`）の図から、採用方式・変更後の構造を示す図を転載する。
  キャプションに「変更後 / 出典: DSN-{CR}」と記載する。

SPO・DSN に該当する図がない、または SP 単位でより詳細な図が必要な場合に限り、下記の判断基準で新規図を追加する。

### 新規図を追加する判断基準（SPO・DSN図で不足する場合のみ）

| 判断基準 | 図種別 | Mermaid 記法 |
|---------|--------|-------------|
| 継承・実装関係を持つクラス階層が変わる | **クラス図** | `classDiagram` |
| 3つ以上の関数をまたぐ呼び出しチェーンが変わる | **関数呼び出し図** | `graph LR` |
| コンポーネント間の処理順序・非同期フローが変わる | **シーケンス図** | `sequenceDiagram` |
| 条件分岐・ループを含む制御フローが変わる | **フローチャート** | `flowchart TD` |
| 状態遷移・ライフサイクルが変わる | **状態遷移図** | `stateDiagram-v2` |
| テーブル・スキーマのリレーションが変わる | **ER図** | `erDiagram` |

### 省略してよいケース（Before/Afterコードで十分）
- フィールド・引数の単純追加（型・名前・デフォルト値が自明）
- 関数シグネチャの変更のみ（呼び出し構造は変わらない）
- ローカル変数の追加・条件式の変更

### 図を追加する場合の方針
- Before/Afterの直後、または「変更箇所の説明」の後に配置する
- 変更前後で構造が変わる場合は Before図・After図を並べる（差分が小さければ After図のみ）
- キャプション例：「図: cmd_list 呼び出しフロー（変更後）」
- スタイリング：変更箇所 `fill:#ffcccc,stroke:#cc0000`、新規追加 `fill:#ccffcc,stroke:#009900`

## R4. エラーハンドリング・フェイルセーフ設計ルール

CHD の After コードは **正常系だけでなく異常系・エラー経路** を必ず含めること。

### 設計に含めるべき観点

| 観点 | 内容 |
|------|------|
| **エラー経路の明示** | After コードに例外・エラーパスを記述する（空の catch や握りつぶしは設計上禁止） |
| **フェイルセーフ方針** | 失敗時の「安全側への倒し方」を設計に明記する（例：デフォルト値を返す、処理をスキップする、etc.） |
| **ロールバック・補償処理** | 状態を変更する操作は、失敗時のロールバック手順または補償処理を設計する |
| **エラーメッセージ設計** | ユーザー向けメッセージとログ/デバッグ向けメッセージを分けて設計する |
| **入力バリデーション** | 外部境界（ユーザー入力・外部APIレスポンス・DB読み出し）でのバリデーション箇所を After コードに含める |

## R3. レビューチェックポイント（品質ゲート）

AIレビュー・人レビューでは以下の観点を必ず確認すること：

### 要求カバレッジ
- [ ] CRS の全 SP が CHD の第3章にいずれかの形で対応しているか
- [ ] Before/After が「設計変更の意図」を説明できているか（コードの複製ではなく）
- [ ] 変更スコープ外のコード変更がないか

### アーキテクチャ準度
- [ ] 採用した DSN の実装方式（`DSN-{CR}.md`）に沿った設計になっているか
- [ ] `project-steering.md` の命名規約・パターンに従っているか
- [ ] モジュール境界・依存方向が既存のアーキテクチャ慣習に従っているか

### 実行可能性
- [ ] Before/After コードが実際に動作する内容か（構文エラーのレベルでよい）
- [ ] エラーハンドリング・バリデーションが After コードに含まれているか（R4 参照）
- [ ] フェイルセーフ方針が明記されているか（失敗時に安全側に倒す設計か）
- [ ] ロールバック・補償処理が必要な操作で設計されているか
- [ ] 変更規模が警告閾値（500行以上の変更）を超える場合、CR分割を提案しているか

### 機械的チェック
- [ ] 第3章の見出しが「SP-{番号}：{仕様名}」形式になっているか
- [ ] トレーサビリティマトリクス（第4章）が全SP × 全変更ファイル × 全変更関数で埋まっているか
- [ ] 変更対象ファイル一覧（第2章）と第3章の記述が一致しているか
- [ ] 図の追加が「R2の判断基準に該当する場合のみ」になっているか
