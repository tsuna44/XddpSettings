# XDDP Step 05 設計品質ルール（DSN: 実装方式検討メモ）

> このファイルは `xddp.05.arch` スキルが参照する設計品質ゲート定義。
> ルールのみを変更する場合はこのファイルだけを編集する。

---

## R1. 実装方式候補の検討ルール

セクション「2. 実装方式の候補」では、実質的に検討に値する案がある場合のみ複数案を挙げること。
アプローチが自明な場合（変更箇所・方法に選択の余地がほぼない場合）は **1案でよい**。
複数案を挙げる場合は以下の軸を意識して **実質的に異なるアプローチ** にすること：

| 軸 | 観点 |
|----|------|
| **Where**（変更を吸収する場所） | 呼び出し元 / 呼び出し先 / 新レイヤー挿入 |
| **Depth**（変更の深さ） | 最小変更 / 部分リファクタリング / 抜本的再設計 |
| **Coupling**（依存の持ち方） | 既存拡張 / 新規独立 / 共通化 |
| **When**（処理タイミング） | 同期 / 非同期・イベント駆動 / バッチ |
| **How**（技術・パターン） | 既存スタック / 新パターン / OSS活用 |

案の名称は内容を端的に表す名前にする（例：「既存クラス拡張案」「新規サービス分離案」「ミドルウェア活用案」）。  
各案の「システム全体への影響」欄に、他モジュール・インターフェースへの波及を記述すること。

## R2. QCD 比較表のルール

セクション「3. 方式比較（QCD）」の Q / C / D 各表をすべての案の列で埋めること。  
評価は `◎ / ○ / △ / ✕` で記入し、根拠を採用方式セクションに反映すること。

---

## R3. システム全体影響コンテキストのルール

セクション「1.1 システム全体への影響コンテキスト」を **必ず** 埋めること。  
CRS・SPO を読み、以下を把握した上で各案を検討すること：
- 関連モジュール・データフロー
- 既存 API への影響
- 非機能要件への影響

## R4. 図の作成ルール（Mermaid）

### SPO図の活用（優先）

SPO（スペックアウト成果物）に作成された図のうち、変更箇所・変更が影響する箇所を示すものを
DSN の「1.1 システム全体への影響コンテキスト」または各案の「システム全体への影響」欄に転載すること。

転載元の候補：
- モジュール個別 SPO（`modules/*-spo.md`）: 状態遷移図・クラス図・処理フロー
- クロスモジュール SPO（`cross-module/SPO-{CR}-cross.md`）: 構造図・シーケンス図

転載する図にはキャプションに「出典: SPO-{CR}（{モジュール名またはクロスモジュール}）」と記載する。
SPO に該当する図がない、または補足が必要な場合に限り、以下の基準で新規図を作成する。

### 新規図の作成（SPO図で不足する場合）

このCRの変更内容を分析し、理解に最も効果的な図を選んで Mermaid で作成すること。  
図は「1.1 システム全体への影響コンテキスト」または各案の「システム全体への影響」欄に配置する。

### 図種別の選択基準

| 変更の性質 | 推奨図種別 | Mermaid 記法 |
|-----------|-----------|-------------|
| 複数モジュール・クラス・ファイルに影響が波及する | **影響範囲図**（モジュール依存グラフ） | `graph LR` / `graph TD` |
| データがどこで生成・変換・蓄積されるか示したい | **DFD**（データフロー図） | `graph LR`（DFD記法） |
| コンポーネント間の呼び出し順序・通信フローが変わる | **シーケンス図** | `sequenceDiagram` |
| データモデル・クラス構造が変わる | **クラス図** | `classDiagram` |
| 処理の分岐・ループ・アルゴリズムが変わる | **フローチャート** | `flowchart TD` |
| 状態遷移・ライフサイクルが変わる | **状態遷移図** | `stateDiagram-v2` |
| DBスキーマ・テーブル構造が変わる | **ER図** | `erDiagram` |
| タスク・フェーズの前後関係が変わる | **タイムライン / ガントチャート** | `timeline` / `gantt` |

### スタイリング規則（影響範囲図の場合）
```
classDef changed   fill:#ffcccc,stroke:#cc0000   ← 変更あり（影響あり）
classDef unchanged fill:#eeeeee,stroke:#999999   ← 変更なし
classDef added     fill:#ccffcc,stroke:#009900   ← 新規追加
```

### DFD 記法ガイド（Mermaid `graph LR` で表現）

| DFD要素 | Mermaid表現 | 例 |
|--------|------------|-----|
| 外部エンティティ | 角括弧ノード `[ ]` | `User["ユーザー"]` |
| プロセス（処理・変換） | 丸括弧ノード `( )` | `P1("1.0 優先度バリデーション")` |
| データストア | 二重角括弧ノード `[[ ]]` | `DS1[["tasks.json"]]` |
| データフロー（矢印） | ラベル付き矢印 `-->` | `User -->|"add title"| P1` |

- プロセスIDは `1.0`, `2.0` のように番号を付ける
- データフローのラベルにデータ名・パラメータ名を記述する
- 変更が発生するプロセス・ストアには上記スタイリング規則を適用する

### 判断フロー
1. CRS・SPO から変更の中心的な性質を特定する
2. 上記表から該当する図種別を選ぶ（変更なしなら図は省略してよい）
3. 複数の性質が混在する場合は最大3枚まで組み合わせる
4. 図ごとにキャプション（例：「図1: モジュール影響範囲」）を付ける

## R5. レビューチェックポイント（品質ゲート）

AIレビュー・人レビューでは以下の観点を必ず確認すること：

### 要求カバレッジ
- [ ] CRS の全 UR に対して、採用方式がカバーしているか
- [ ] SR / SP レベルの制約・非機能要件が方式に反映されているか
- [ ] SPO で発見した影響モジュールがすべて考慮されているか

### アーキテクチャ準度
- [ ] 採用案が既存アーキテクチャの慣習・パターンと整合しているか（`project-steering.md` 参照）
- [ ] モジュール境界を不必要に越えていないか
- [ ] 循環依存・過剰な結合が発生していないか

### 実行可能性
- [ ] 採用案の実装難易度が現チームのスキルセットで現実的か
- [ ] 既存コードへの影響範囲が想定内か（`SPECOUT_MAX_AFFECTED_FILES` 以内か）
- [ ] 非機能要件（性能・セキュリティ・可用性）を満たせるか

### 機械的チェック
- [ ] 複数案が存在する場合、実質的に異なるアプローチか（自明な場合は1案でよい）
- [ ] QCD 表が全案・全項目で埋まっているか
- [ ] 図が 1〜3 枚作成されているか（または不要と明記されているか）
- [ ] 採用方式の根拠が定量的・具体的に記述されているか
