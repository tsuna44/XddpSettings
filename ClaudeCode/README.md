# XDDP派生開発 成果物構成

## プロジェクトフォルダ構成

```
project-root/
├── CLAUDE.md                              # プロジェクト固有ルール（常時読み込み）
│
├── .claude/skills/
│   ├── xddp-requirements/SKILL.md        # 要求分析
│   ├── xddp-specout-feedback/SKILL.md    # スペックアウト→仕様書フィードバック
│   ├── xddp-conflict-check/SKILL.md      # 並行CR衝突検知
│   ├── xddp-coding/
│   │   ├── SKILL.md                      # コーディング
│   │   └── coding-rules.md              # 共通コーディングルール（オンデマンド）
│   ├── xddp-verify/SKILL.md              # 変更後静的確認
│   ├── xddp-testcase/SKILL.md            # テストケース生成
│   └── xddp-test-feedback/SKILL.md       # テストNG→変更設計書フィードバック
│
└── CR-YYYY-NNN/                           # CRごとのフォルダ
    ├── CR-YYYY-NNN.md                     # 変更要求書
    ├── 変更要求仕様書-CRS-YYYY-NNN.md     # 変更要求仕様書（Markdown）
    ├── 変更要求仕様書-CRS-YYYY-NNN.xlsx   # 変更要求仕様書（Excel）
    ├── スペックアウト-CR-YYYY-NNN.md      # スペックアウト資料
    ├── TM-CR-YYYY-NNN.xlsx                # TM（Excel専用）
    ├── 変更設計書-CR-YYYY-NNN.md          # 変更設計書
    ├── テストケース-CR-YYYY-NNN.md        # テストケース仕様書（Markdown）
    ├── テストケース-CR-YYYY-NNN.xlsx      # テストケース仕様書（Excel）
    ├── テスト結果記録-CR-YYYY-NNN.md      # テスト結果記録
    ├── 衝突チェック-CR-YYYY-NNN.md        # 衝突チェック記録
    └── 修正ループ管理-CR-YYYY-NNN.md      # テストNG修正ループ管理
```

## 各成果物の役割と作成タイミング

| # | 成果物 | 作成者 | 作成タイミング | レビュアー |
|---|--------|--------|-------------|-----------|
| 1 | CR（変更要求書） | 顧客 or 受注側 | 顧客依頼受領時 | - |
| 2 | 変更要求仕様書 | 設計担当者 | CR受領後・ソース調査前に骨子、調査後に確定 | 仕様の理解者・母体の有識者 |
| 3 | スペックアウト資料 | 設計担当者 | 変更要求仕様書の骨子作成後 | 設計担当者 |
| 4 | TM | 設計担当者 | スペックアウト後（ソースファイル確定後） | 母体の有識者・設計担当者 |
| 5 | 変更設計書 | 設計担当者 | TM完成後 | 母体の有識者・設計担当者 |

## 文書間のトレーサビリティ

```
CR
  └─ 変更要求仕様書（仕様番号：CRS-YYYY-NNN-XX-XX）
        ├─ 由来列：CR／スペックアウト
        └─ スペックアウト資料
              ├─ フィードバック先仕様番号 → 変更要求仕様書
              └─ TM（Excel）
                    ├─ 行：仕様番号（変更要求仕様書と紐付け）
                    ├─ 列：ソースファイル（スペックアウト資料と紐付け）
                    └─ 変更設計書
                          ├─ TM交点ごとに1つ作成
                          └─ テストNG発生時に修正エントリを追加
                                ↑
                          テスト結果記録（NG）
                          ← xddp-test-feedback がフィードバック
```

## テストNG修正ループ

```
テスト実施
  ↓ NGあり
xddp-test-feedback（原因分類・修正エントリ生成）
  ↓
  ├─ 原因A/B → 変更設計書に修正エントリ追加
  │     ↓ xddp-coding（修正実装）
  │     ↓ xddp-verify（静的確認）
  │     ↓ テスト再実施
  │     └─ OK → ループ終了
  │
  ├─ 原因C → 変更要求仕様書を修正（要レビュー承認）
  │           ↓ 変更設計書を更新
  │           ↓ xddp-coding → テスト再実施
  │
  └─ 原因D → テストケースを修正して再実施

※ 3回を超える場合はCR全体の見直しを推奨
```

## ステータス管理

各成果物のステータスは以下の4段階で管理する：

| ステータス | 意味 |
|-----------|------|
| 作成中 | 記述中。レビュー未実施 |
| レビュー待ち | 記述完了。レビュー依頼済み |
| 承認済み | レビュー完了・承認済み |
| 却下 | レビューで却下。要修正 |

## Skillsの配置と役割

| Skill | 起動トリガー例 | 入力 | 出力 |
|-------|-------------|------|------|
| xddp-requirements | 「このCRを分析して」 | CR | 変更要求仕様書への追記内容 |
| xddp-specout-feedback | 「スペックアウトの結果を仕様書に反映して」 | スペックアウト資料＋変更要求仕様書 | 両ファイルの更新 |
| xddp-conflict-check | 「衝突チェックして」 | 全並行CRのTM＋変更要求仕様書＋変更設計書 | 衝突検知レポート |
| xddp-tm-generate | 「変更設計書からTMを生成して」 | 変更設計書＋スペックアウト資料＋変更要求仕様書 | TM（Markdown＋Excel） |
| xddp-coding | 「変更設計書に基づいてコードを変更して」 | 変更設計書＋ソースファイル | 変更済みソースファイル |
| xddp-verify | 「確認項目をチェックして」 | 変更設計書＋変更済みソースファイル | 検証レポート＋チェックリスト |
| xddp-testcase | 「テストケースを生成して」 | 変更設計書＋スペックアウト資料＋変更要求仕様書 | テストケース仕様書（Markdown） |
| xddp-test-feedback | 「テストNGを変更設計書に反映して」 | テスト結果記録＋変更設計書＋変更要求仕様書 | 修正エントリ追加＋フィードバックレポート |

### 共通コーディングルールの配置

```
.claude/skills/xddp-coding/
├── SKILL.md           # Skillの定義（起動条件・手順）
└── coding-rules.md    # 全プロジェクト共通コーディングルール
                       # ← CLAUDE.mdには書かず、xddp-coding実行時にのみ読み込む
```

**設計の意図**：コーディングルールはコード変更時にだけ必要なため、
CLAUDE.md（常時読み込み）ではなくSkillのサポートファイルとして配置する。
プロジェクトのCLAUDE.mdにはCR番号・ベースライン情報など
プロジェクト固有の情報のみを記載する。
