## 発展候補

### 1. 過去CR類似パターン検索（RAG）

**概要**：過去のCR成果物をベクトルDBに蓄積し、新しいCR起票時に類似事例を自動提示する。

**必要環境**：
- Python 3.9〜3.12
- ChromaDB（ローカルベクトルDB）
- sentence-transformers（`paraphrase-multilingual-mpnet-base-v2`・日本語対応）
- SQLite 3.35以上

**検索対象**：
- CRのタイトル・背景・ウォンツ
- 変更要求仕様書の仕様番号・Before/After
- 変更設計書の修正方針・変更対象
- テストケースの観点・確認内容

**期待効果**：
- 過去に同種の変更を行った際の設計ノウハウを再利用できる
- 類似CRで発生したNGパターンを事前に把握して品質を向上できる
- 新規担当者でも過去事例を参照しながら作業できる

**実装ファイル構成（予定）**：
```
.claude/skills/xddp-search/
├── SKILL.md              # 「類似CR検索して」で起動
├── xddp_indexer.py       # 成果物をパースしてChromaDBに登録
├── xddp_searcher.py      # クエリを受けて類似CRを返す
└── requirements.txt      # chromadb, sentence-transformers
```

### 2. 変更要求仕様書→TM→変更設計書の一貫自動生成

**概要**：
変更要求仕様書が承認された時点で、TM・変更設計書の骨子を自動生成する。
現在はTM生成（xddp-tm-generate）のみ実装済み。

**未実装部分**：
- 変更設計書の骨子自動生成（ヘッダ・修正方針の空欄を埋める）
- 変更設計書→テストケースの自動連鎖生成

**期待効果**：
- CR受領からコーディング開始まで一気通貫で骨子が揃う
- 担当者は内容の確認・補完だけに集中できる

### 3. ベースラインバージョン管理との連携

**概要**：
Gitのコミット情報とCR番号・仕様番号を紐付け、
「このコミットがどのCR・どの仕様に対応するか」を自動でトレースする。

**実装イメージ**：
- コミットメッセージの`[CR-YYYY-NNN]`タグをパースしてDBに記録
- `git log`と変更要求仕様書を照合して未コミットの仕様を検出
- リリース時に「どのCRが含まれているか」を自動でリリースノートに出力

**期待効果**：
- コードレビュー時にその変更が何の仕様に対応するかが即座にわかる
- リリース前の「仕様漏れチェック」が自動化できる

---

### 4. レビュー支援エージェント

**概要**：
変更要求仕様書・変更設計書・コードの3点を突き合わせて
レビュー時の指摘候補を自動生成するエージェント。

**チェック項目（案）**：
- 仕様書のBefore/Afterとコードの変更が一致しているか
- 変更設計書の確認項目がテストケースに網羅されているか
- ADD/MOD/DELタグが変更設計書の全エントリに対応しているか
- 並行CRとの衝突が新たに生じていないか（xddp-conflict-checkの自動実行）

**期待効果**：
- レビュアーの確認漏れを防ぐ
- 「設計書とコードのズレ」を人間が見る前にAIが検出できる

### 5. ダッシュボード（進捗・品質可視化）

**概要**：
全CRの進捗・テスト結果・修正ループ回数をリアルタイムで可視化する
Webダッシュボード。

**表示項目（案）**：
- CR一覧とフェーズ別の進捗状況

- 並行CR数と衝突検出件数
- 仕様ステータス（作成中／レビュー待ち／承認済み）の分布

**実装候補技術**：
- Streamlit（Pythonだけで構築できる・シンプル）
- Markdownファイルをパースしてリアルタイム集計

## 大規模コードベースへの対応（複数リポジトリ・400kstep以上）

### 7. 静的解析ツールとの連携による呼び出しグラフ自動生成

**背景**：
400kstep以上・複数リポジトリの環境では、手動のgrepによる影響範囲特定に限界がある。
現在のxddp-specout Skillに組み込んだ波紋検索・打ち切り基準を補完するために、
静的解析ツールで呼び出しグラフを自動生成し、スペックアウト資料のシーケンス図・DFDの
元データとして活用する。

**実装イメージ**：
- `cflow`（C言語）や`doxygen`で呼び出しグラフをGraphviz形式で出力
- グラフをパースしてスペックアウト資料のMermaidシーケンス図に変換するSkillを追加
- 変更対象識別子を起点に、波紋検索の各波をグラフ上でトレースして自動抽出

**期待効果**：
- スペックアウトの調査時間を大幅短縮
- 人間が見落としやすい間接参照を網羅的に検出できる
- 大規模コードベースでも一定の品質を保った影響範囲の特定が可能

**対応ツール候補**：

| ツール | 言語 | 用途 |
|--------|------|------|
| `cflow` | C/C++ | 関数呼び出しツリーのテキスト出力 |
| `doxygen` | C/C++/Python | 呼び出しグラフをGraphvizで可視化 |
| `ast-grep` | 多言語 | ASTベースのパターン検索 |
| `ctags` + `cscope` | C/C++ | 関数定義・参照の一覧化 |
| `understand` | 多言語 | 商用・大規模向け静的解析 |

### 8. チェンジセットマップの蓄積と再利用

**背景**：
大規模コードベースでは、過去に調査した影響範囲の知識を再利用できれば
2回目以降の類似変更の調査コストが大幅に下がる。

**実装イメージ**：
- スペックアウト資料の打ち切り記録を構造化してYAMLファイルに蓄積する
- 「この識別子を変更した場合の影響範囲・打ち切り箇所」をプロジェクト固有の知識として管理する
- 新しいCRで同じ識別子が登場した際に過去の調査結果を自動提示する（xddp-specout Skillの基準5と連動）

```yaml
# change_impact_map.yaml（蓄積例）
SENSOR_THRESHOLD:
  最終調査CR: CR-2024-031
  直接参照: [sensor_detect.c:105, sensor_state.c:88]
  打ち切り箇所:
    - 場所: monitor_loop.c
      基準: 基準1（引数・戻り値の型が変わらない）
    - 場所: uart_send.c
      基準: 基準3（送信フォーマット変更なし）
  影響なし確認済み: [ui_layer/, network_layer/]
```

**期待効果**：
- 2回目以降の同種変更で調査工数をほぼゼロにできる
- 「なぜこの箇所を調査しなかったか」の根拠が蓄積される
- 新規メンバーがプロジェクト固有の影響範囲の知識を素早く習得できる

**RAGとの連携（発展項目1と組み合わせ）**：
- チェンジセットマップをRAGのインデックスに追加することで
  「この識別子の過去の影響範囲は？」という自然言語クエリに答えられる

## RAG検索基盤の技術選定

### DB選択フロー

```
使う人数・環境
  ↓
  1人 or 同一PC内で完結
    ↓
    意味検索が必要か？
      不要（用語が統一されている）→ SQLite + FTS5
      必要（担当者多い・用語バラバラ）→ ChromaDB 組み込みモード
  ↓
  複数人が同時アクセスする
    → ChromaDB サーバーモード（Dockerが楽）

オフライン環境（ネット非接続）
  → SQLite 一択（モデルダウンロード不要）
  → ChromaDB組み込みも可（初回モデルDL後はオフライン動作）
```

### SQLite + FTS5 vs ChromaDB 技術比較

| 項目 | SQLite + FTS5 | ChromaDB 組み込み | ChromaDB サーバー |
|------|:------------:|:----------------:|:----------------:|
| **サーバープロセス** | 不要 | 不要 | 必要（常駐） |
| **複数プロセス同時アクセス** | ✅ | ❌ | ✅ |
| **追加インストール** | 不要 | chromadb | chromadb + Docker |
| **埋め込みモデル** | 不要 | 必要（約400MB） | 必要（約400MB） |
| **Python バージョン** | 3.6以上 | 3.9〜3.12 | 3.9〜3.12 |
| **SQLite バージョン** | 標準のまま | 3.35以上 | 3.35以上 |
| **RAM（検索時）** | 数MB | ベクトル数×次元×4B | ベクトル数×次元×4B |
| **GPU** | 不要 | 不要（あれば高速） | 不要（あれば高速） |
| **初回起動時間** | 即時 | 数十秒（モデルロード） | 数十秒（モデルロード） |
| **検索精度** | キーワード一致 | 意味的類似度 | 意味的類似度 |
| **バージョン管理** | Python標準 | マイグレーション不可逆 | マイグレーション不可逆 |
| **認証・アクセス制御** | ファイル権限 | なし（v1.0.x） | 別途設定が必要 |
| **オフライン動作** | ✅ 完全 | ✅（初回DL後） | ✅（初回DL後） |

### 推奨移行パス

```
フェーズ1（今すぐ）
  SQLite + FTS5
  → 追加インストールなし・即日導入可能

フェーズ2（CR蓄積後・意味検索が欲しくなったら）
  sqlite-vec を追加
  → FTS5と共存・ChromaDB不要のまま意味検索が追加できる

フェーズ3（チーム全体で共有したくなったら）
  ChromaDB サーバーモード（Docker）に移行
  → 複数人同時アクセス対応
```
