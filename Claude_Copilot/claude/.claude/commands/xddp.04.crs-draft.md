# XDDP 変更要求仕様書ドラフト生成

## 目的
xddp.01.requirementsの要求分析結果を入力として、
USDM形式の変更要求仕様書（Markdown）ドラフトを生成する。
このドラフトはxddp.03.crs-ai-reviewでAIレビューを受けた後、人間レビューへ進む。

## 入力
- 要求分析結果（xddp.01.requirementsの出力）
- 変更要求仕様書テンプレート（`01_テンプレート/変更要求仕様書-template.md`）

## 手順

### Step 1: 要求分析結果の読み込み
要求分析結果から以下を抽出する：
- ウォンツ → ユーザ要求（UR）に変換
- ニーズ → システム要求に変換
- 制約 → 制約・前提条件セクションに記載
- 懸念事項 → 確認事項・未解決事項セクションに記載

### Step 2: USDM階層の構築
以下の3階層で仕様を構造化する：

```
カテゴリ
  └─ ユーザ要求（UR-NN）
       └─ システム要求（CRS-YYYY-NNN-NN）
            └─ 仕様（CRS-YYYY-NNN-NN-NN）
                 ├─ Before：変更前の状態
                 └─ After ：変更後の状態
```

#### カテゴリの判断基準

カテゴリ定義ファイルが存在する場合は必ずそちらを優先する：
`~/claude/xddp/categories.md`（プロジェクト共通）
またはプロジェクトルートの `.xddp/categories.md`（プロジェクト固有）

定義ファイルが存在しない場合はデフォルト定義を使用する：

| カテゴリ | 配置する仕様の基準 | 配置しない例 |
|---------|----------------|------------|
| **機能** | システムの動作・振る舞い・データ処理を変える仕様。センサー・アクチュエータ・通信・演算ロジック等 | テスト手順、ドキュメント更新 |
| **デバッグツール** | 製品出荷時には無効化されるデバッグ・開発支援機能に関する仕様。ログ出力・診断コマンド等 | 製品機能として常時動作するもの |
| **テスト** | テストケース・テスト手順・テスト環境・テストツールに関する仕様 | 製品の機能仕様 |
| **日程** | リリース日・納期・マイルストーン・作業スケジュールに関する仕様 | 機能要件 |
| **成果物管理** | 設計書・仕様書・ソースコード・バイナリなどの成果物の管理・提出・形式に関する仕様 | 製品の動作仕様 |

**判断が迷う場合のルール：**
- 「製品を動かすと変わる動作」→ 機能
- 「テストのときだけ使う」→ テスト
- 「書類・ファイルの話」→ 成果物管理
- 「いつリリースするかの話」→ 日程
- それでも判断できない場合は「機能」に仮配置して `（要確認：カテゴリ）` と注記する

### Step 3: 仕様番号の採番
以下のルールで仕様番号を採番する：
- CR番号から`CR-YYYY-NNN`を取得
- ユーザ要求：`UR-01`〜順番に採番
- システム要求：`CRS-YYYY-NNN-01`〜順番に採番
- 仕様：`CRS-YYYY-NNN-01-01`〜順番に採番
- 由来列：CRに明記されているものは`CR`、推定・追加が必要なものは`作成中`

### Step 4: ドラフト生成
変更要求仕様書テンプレートに沿ってMarkdownを生成する。
不明・未確定の箇所は`（要確認：〇〇）`と明示して空欄にしない。

### Step 5: 出力
以下のファイル名でMarkdownを出力する：
```
変更要求仕様書-CRS-YYYY-NNN.md
```

生成後に以下のサマリを表示する：
```
## ドラフト生成サマリ

| 項目 | 件数 |
|------|------|
| ユーザ要求（UR） | N件 |
| システム要求 | N件 |
| 仕様 | N件 |
| 要確認項目 | N件 |

→ 次のステップ：/xddp.03.crs-ai-review で AIレビューを実施してください
```

## 注意事項
- 推測で仕様を追加しない。要求分析結果に明記されている内容のみ記述する
- Before/Afterが不明な場合は`（要確認）`と記載し、空欄にしない
- 1つのユーザ要求に対して複数のシステム要求が必要な場合は分割して記述する
- 仕様のAfterが「変更なし」の場合も明示的に記述する（影響確認の証跡として残す）
