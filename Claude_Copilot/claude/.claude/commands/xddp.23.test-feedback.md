# XDDP テストNG フィードバックSkill

## 目的
テストNGを起点として変更設計書に修正エントリを追加し、
コーディング→テストの修正ループを品質記録として残しながら回す。

## 入力
- テスト結果記録ファイル（NGが記録されたもの）
- 変更設計書ファイル（修正エントリを追記する対象）
- 変更要求仕様書ファイル（仕様側の問題か判定するため）
- スペックアウト資料ファイル（実装の誤り分析のため）

## NGの原因分類

NGが発生した場合、まず以下の4種類に原因を分類する：

| 原因種別 | 定義 | 対応先 |
|---------|------|--------|
| A: 実装ミス | コードが変更設計書通りに実装されていない | 変更設計書に修正エントリ追加 |
| B: 設計ミス | 変更設計書の内容自体が誤っている | 変更設計書を修正＋**TMを更新**（変更箇所の増減を反映） |
| C: 仕様ミス | 変更要求仕様書のBefore/Afterが誤っている | 変更要求仕様書を修正（要レビュー承認）＋**TMを更新**＋変更設計書を更新 |
| D: テスト仕様ミス | テストケースの手順・期待結果が誤っている | テストケースを修正して再実施 |

## 手順

### Step 1: NGの収集
テスト結果記録から確認結果が「NG」の行を全件抽出する。

### Step 2: 原因分類
各NGについて以下の判定フローで原因を分類する：

```
テストNG発生
  ↓
変更設計書の確認項目と照合
  ├─ 確認項目通りの結果が出ていない
  │    ↓
  │   コードと変更設計書を照合
  │    ├─ コードが設計書通りでない → 原因A（実装ミス）
  │    └─ コードが設計書通りだが結果がNG
  │         ↓
  │        変更設計書の変更方針を確認
  │         ├─ 設計方針が誤っている → 原因B（設計ミス）
  │         └─ 設計方針は正しいが仕様のBefore/Afterが誤り → 原因C（仕様ミス）
  └─ 確認項目の手順・期待結果が不正確 → 原因D（テスト仕様ミス）
```

### Step 3: 変更設計書への修正エントリ追加
原因A・Bの場合、変更設計書に以下の修正エントリを追加する：

```markdown
## 修正エントリ：{TC番号} NG対応 × {ファイル名}

### ヘッダ
| 項目 | 内容 |
|------|------|
| 起因TC番号 | TC-XX-NNN-XX |
| NG内容 | （何がNGだったか） |
| 原因種別 | A: 実装ミス／B: 設計ミス |
| 対応仕様番号 | CRS-NNN-XX-XX |
| ソースファイル | （ファイル名） |
| 変更対象 | （関数名／定数名） |
| ステータス | 作成中 |
| 記述者 | （氏名） |
| 日付 | YYYY-MM-DD |

### NG内容と原因分析
（何がNGで、なぜそうなったかを文章で記述）

### 修正方針
（どのように修正するか。設計の意図を残す）

### 関数内の変更
**関数名**：（関数名）　変更種別：■変更 □追加 □削除

| No | 変更内容 | 予想行数 |
|----|---------|---------|
| 1 | （修正内容を文章で記述） | （行数） |

### 変更後の確認項目
| No | 確認内容 | 期待結果 |
|----|---------|---------|
| 1 | （NGだったケースを再確認） | （正しい期待結果） |
| 2 | （修正による影響確認） | （期待結果） |
```

### Step 4: 変更要求仕様書への影響判定
原因Cの場合、変更要求仕様書の該当仕様のステータスを「却下」に変更し、
修正した仕様を新規行として追加する指示を出力する。
**必ずレビュアーへ通知し、仕様変更として承認を得てから次のステップへ進む。**

### Step 5: TMへの影響判定
原因B・Cの場合、TMの更新要否を判定して出力する：

**原因Bの場合（設計ミス）：**
変更設計書の修正内容を確認し、以下の変化があればTMを更新する：
- 変更対象ファイルが増減した場合 → 該当仕様番号の列（●）を追加・削除
- 変更対象ファイルは同じだが変更種別が変わった場合 → ●のまま（TMは変更不要）

**原因Cの場合（仕様ミス）：**
変更要求仕様書の修正内容を確認し、以下の変化があればTMを更新する：
- 仕様番号が変わった場合 → TMの行を更新
- 変更対象ファイルが増減した場合 → 該当行の●を追加・削除
- 仕様が削除された場合 → TMから該当行を削除

更新が必要な場合は `/xddp.11.tm-generate` を再実行してTMを再生成することを推奨する。

### Step 6: テストケースへの影響判定
原因Dの場合、修正が必要なテストケースを特定して出力する。

### Step 6: フィードバックレポートの出力

```markdown
## テストNG フィードバックレポート

**対応CR番号**：CR-YYYY-NNN
**レポート日時**：YYYY-MM-DD
**NGの総数**：N件

---

### NG一覧と原因分類

| TC番号 | テストケース名 | NG内容 | 原因種別 | 対応先 |
|--------|-------------|--------|---------|--------|
| TC-WB-031-02 | しきい値でアラーム発生 | 戻り値がfalseだった | A: 実装ミス | 変更設計書に修正エントリ追加 |

---

### 変更設計書への追記内容
→ 修正エントリ：TC-WB-031-02 NG対応 × sensor_detect.c を追加

### 変更要求仕様書への影響
→ 影響なし／CRS-NNN-XX-XXのステータスを却下に変更（要レビュー承認）

### TMへの影響
→ 影響なし／仕様番号CRS-NNN-XX-XXの行を更新（原因B・Cの場合）
→ 更新が必要な場合は /xddp.11.tm-generate を再実行してTMを再生成する

### テストケースへの影響
→ 影響なし／TC-XX-NNN-XXの期待結果を修正

---

### 修正後の実行手順

**原因A（実装ミス）の場合：**
1. `/xddp.16.coding` で修正エントリをソースに反映する
2. `/xddp.17.verify` で自己確認を実施する
3. `/xddp.18.coding-ai-review` でAIレビューを実施する
4. テストを再実施してOKになれば本ループを終了する

**原因B（設計ミス）の場合：**
1. `/xddp.14.design-review-fix` で変更設計書を修正する
2. TMの更新が必要な場合は `/xddp.11.tm-generate` を再実行する
3. `/xddp.16.coding` でコードを修正する
4. `/xddp.17.verify` → `/xddp.18.coding-ai-review` でレビューを実施する
5. テストを再実施する

**原因C（仕様ミス）の場合：**
1. レビュアーへ通知し、仕様変更の承認を得る
2. `/xddp.05.crs-ai-review` で変更要求仕様書の修正をレビューする
3. TMの更新が必要な場合は `/xddp.11.tm-generate` を再実行する
4. `/xddp.14.design-review-fix` で変更設計書を更新する
5. `/xddp.16.coding` でコードを修正する
6. テストを再実施する

**原因D（テスト仕様ミス）の場合：**
1. `/xddp.22.testcase-review-fix` でテストケースを修正する
2. テストを再実施する

### 修正ループ履歴

| ループ回数 | 日付 | NG件数 | 原因種別 | 対応状況 |
|-----------|------|--------|---------|---------|
| 1回目 | YYYY-MM-DD | N件 | A: N件, B: N件, C: N件, D: N件 | 対応中 |
```

## 注意事項
- 原因Cが検出された場合は必ずレビュアーへ通知し、仕様変更として承認を得てから
  変更設計書・コードの修正に進む
- 原因B・Cでは必ずTMへの影響を確認する。影響がある場合は `/xddp.11.tm-generate` を再実行する
- 修正ループは原則3回以内を目安とする。3回を超える場合はCR全体の見直しを推奨する
- 修正エントリには必ず「起因TC番号」を記載し、どのNGへの対応かをトレースできる状態にする
- 変更設計書の修正エントリ追加後は `/xddp.15.conflict-check` を再実行して
  新たな衝突が生じていないかを確認する
