# XDDP システム仕様書更新

## 目的
テスト完了・承認済みのCR変更内容を反映し、
変更後のシステム全体の仕様書を最新状態に更新する。
次回のCR・派生開発のベースラインとなる文書を生成する。

## 前提条件
- 全テストケースがOK（承認済み）であること
- xddp.23.test-feedbackのループが終了していること

## 入力
- 既存のシステム仕様書（ベースライン・存在する場合）
- 変更要求仕様書（今回のCRで何が変わったか）
- スペックアウト資料（変更後のシステム構成の根拠）
- 変更設計書（変更内容の詳細）
- テスト結果記録（変更が正しく反映されていることの証跡）

## 出力ファイル構成

```
docs/spec/
├── spec-system-overview.md      # システム概要・全体構成
├── spec-requirements.md         # システム要求一覧（全CR蓄積）
├── spec-functions.md            # 機能仕様一覧
├── spec-interfaces.md           # 外部インタフェース仕様
└── spec-constraints.md          # 制約・前提条件一覧
```

## 手順

### Step 1: 変更箇所の特定
変更要求仕様書・変更設計書から今回の変更内容を整理する：
- 変更・追加・削除された機能
- 変更された定数・データ構造
- 変更された外部インタフェース

### Step 2: 既存仕様書との差分適用
既存の仕様書が存在する場合、変更箇所を特定して更新する。
存在しない場合はスペックアウト資料・変更設計書から全体を新規生成する。

各ファイルの更新内容：

**spec-system-overview.md**
- システム全体の機能ブロック図（Mermaid）
- 主要モジュール一覧と役割
- 変更履歴（CR番号・変更概要・日付）

**spec-requirements.md**
- 全仕様番号と仕様内容の一覧
- 変更状態（最新のAfterを正として記載）
- 由来CR番号（どのCRで追加・変更されたか）

**spec-functions.md**
- 機能ごとの仕様記述（変更後の状態）
- 入力・処理・出力の概要
- 関連する定数・データ構造

**spec-interfaces.md**
- 外部IF仕様（変更後の状態）
- データフォーマット・通信プロトコル

**spec-constraints.md**
- 制約・前提条件一覧（変更後の状態）
- 安全規格・性能要件

### Step 3: 変更履歴の更新
各ファイルの変更履歴テーブルに以下を追記する：

| 版 | 日付 | CR番号 | 変更概要 | 変更者 |
|----|------|--------|---------|--------|

### Step 4: 出力サマリ

```markdown
## システム仕様書更新サマリ

**対応CR番号**：CR-YYYY-NNN
**更新日**：YYYY-MM-DD

| ファイル | 更新内容 | 更新箇所数 |
|---------|---------|----------|
| spec-system-overview.md | （更新内容の概要） | N箇所 |
| spec-requirements.md | 仕様NNN件を更新 | N件 |
| spec-functions.md | （更新内容の概要） | N箇所 |
| spec-interfaces.md | （更新内容の概要） | N箇所 |
| spec-constraints.md | （更新内容の概要） | N箇所 |

→ 次のステップ：/xddp.25.design-update で設計書を更新してください
```

## 注意事項
- 変更箇所以外の既存記述は変更しない（確認した上で維持する）
- 「変更後の状態」を正として記載する（Before/Afterの差分形式にしない）
- 既存の仕様書がない場合は本コマンドが初版となる（初版は全体を新規生成する）
- スペックアウト資料で「打ち切り」とした範囲も、既存記述があれば維持する
