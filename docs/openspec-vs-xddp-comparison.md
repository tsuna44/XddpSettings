# OpenSpec vs XDDP 比較分析

作成日: 2026-04-22  
更新日: 2026-04-26（OpenSpec v1.3.1 時点の情報に更新）

| ツール | 提供元 | 設計思想 |

| OpenSpec | Fission AI（オープンソース） | 軽量な仕様駆動開発（SDD）、30+ AIツール対応、スピード・反復重視 |
| XDDP | 本リポジトリ | eXtreme Design-Driven Process、日本企業向け詳細工程管理、品質・知見蓄積重視 |

## 1. OpenSpec とは

Fission AI が開発した、AI コーディングアシスタント向けの軽量な仕様駆動開発（Spec-Driven Development）フレームワーク。最新版は **v1.3.1（2026年4月21日リリース）**。

### 4つの設計原則

| 原則 | 内容 |
|------|------|
| **流動的で硬直的でない** | フェーズロックなし。意味のある順序でアーティファクトを作成可能 |
| **反復的でウォーターフォールでない** | 構築しながら学び、進めながら改善 |
| **シンプルで複雑でない** | 軽量セットアップ・最小限の手続き |
| **ブラウンフィールド優先** | 新規ではなく既存コードへの変更を前提とした設計 |

### 対応 AI ツール
Claude Code, Cursor, Windsurf, GitHub Copilot, Amazon Q Developer, Continue, Cline, Gemini CLI, Qwen Code, RooCode, Junie, Lingma IDE, ForgeCode, IBM Bob など **30+ のAIコーディングアシスタント**に対応（v1.3.0 でさらに拡張）。

### 9つのOPSXコマンド（v1.0 以降のアクションベース体系）

| コマンド | 内容 |
|----------|------|
| `/opsx:propose` | 変更を作成し、計画アーティファクトを一度に生成 |
| `/opsx:explore` | アイデアを思考・問題調査・要件明確化 |
| `/opsx:new` | 新しい変更のスキャフォルドを開始（段階的） |
| `/opsx:continue` | 次のアーティファクトを1つ作成 |
| `/opsx:ff` | 計画アーティファクトをまとめて一括生成（fast-forward） |
| `/opsx:apply` | タスクを実装、アーティファクトを更新 |
| `/opsx:verify` | 実装をアーティファクトに対して検証 |
| `/opsx:archive` | 完了した変更をアーカイブ |
| `/opsx:bulk-archive` | 複数の完了した変更をまとめてアーカイブ |

> **v1.0（OPSX）の変更点：** 旧来の「Propose → Apply → Archive の3フェーズ制」を廃止し、アクションベースの柔軟なシステムへ移行。依存関係は進捗を示すが強制しない。

### プロファイルシステム（v1.2.0 以降）

| プロファイル | 含まれるコマンド |
|------------|----------------|
| **core** | propose, explore, apply, archive（デフォルト） |
| **custom** | 独自ワークフロー追加可 |

### 主な成果物

```
openspec/
├── specs/                    ← 仕様・要件・シナリオ（最新版を保持）
└── changes/
    ├── {feature}/
    │   ├── proposal.md       ← なぜこの変更をするのか
    │   ├── specs/            ← 仕様デルタ（ADDED/MODIFIED/REMOVED形式）
    │   ├── design.md         ← 技術的アプローチ
    │   └── tasks.md          ← 実装タスク（チェックリスト形式）
    └── archive/              ← 完了したプロジェクト
```

## 2. プロセス体系の比較

| 観点 | OpenSpec | XDDP |
|------|----------|------|
| フェーズ数 | 9コマンド（アクションベース・順序自由） | 15工程（詳細・厳格） |
| 開発対象スコープ | 単一機能・小〜中規模変更 | 単一変更要求（CR）・大規模変更対応 |
| 自動化度 | 高い（AI がアーティファクトを自動生成） | 中程度（AI生成 → 人間レビュー → 修正の反復） |
| 人間の介入 | 軽量（レビューは optional） | 各工程のマンダトリーレビュー（最大5ループ） |
| 知見管理 | 明示的な仕組みなし | lessons-learned.md + improvement-backlog.md |
| トレーサビリティ | 軽量（仕様デルタで変化を記録） | 強固（UR→SR→SP の3層分解・CHDで一対一対応） |
| 文書形式 | Markdown 中心（柔軟） | Markdown + Excel（USDM形式）・標準化厳密 |
| 国際化 | 30+ AIツール対応・多言語可 | 日本語ベース・特定プロセス向け |
| バージョン管理 | `openspec/specs/` で最新版を保持 | `latest-specs/` でバージョン管理 |
| 設計検証 | `/opsx:verify`（実装とアーティファクトの整合確認） | VERIFY（CHD After コードと実装の乖離チェック） |
| ブラウンフィールド対応 | 明示的な設計思想として組み込み | SPO（母体システム影響調査）で対応 |

## 3. OpenSpec にあって XDDP にないもの

| 機能・特徴 | 説明 | 設計思想 |
|-----------|------|---------|
| 軽量性・迅速性 | ウォーターフォール型ではなく、最小限の計画で開発開始 | スピード・アジャイル重視 |
| 30+ AIツール統一 | Claude Code, Cursor, Copilot など多様なツール対応（v1.3.0時点） | プラットフォーム中立的 |
| Optional なレビュー | レビューを強制しない設計 | スピード・自動化重視 |
| Archive による変更履歴管理 | 完了した変更を archive/ に蓄積 | 変更経緯の軽量なトラッキング |
| プロファイルシステム | core/custom プロファイルで有効コマンドセットを制御 | 段階的採用・カスタマイズ |
| フェーズロックなし（アクションベース） | アーティファクトを任意の順序・タイミングで生成可能 | 探索的開発・アジャイルに適合 |
| `/opsx:explore` | 実装前にアイデアを自由に探索・問題を調査するコマンド | 上流不確実性の早期解消 |
| `/opsx:bulk-archive` | 複数の完了した変更をまとめてアーカイブ | 並行CR管理の効率化 |
| ツール自動検出（v1.2.0） | `.claude/` 等のディレクトリを自動スキャンし設定済みツールを検出 | セットアップ摩擦の最小化 |

> **注（仕様デルタについて）:** OpenSpec の「仕様デルタ（`changes/{feature}/specs/`）」は ADDED/MODIFIED/REMOVED 形式で変更内容を記述する。XDDPの CRS（変更要求仕様書）の Before/After 列と**実質的に同じ概念**。XDDPは UR→SR→SP の3層階層に紐づいた表形式で管理しており、構造化の程度が異なる。

> **注（検証コマンドについて）:** OpenSpec の `/opsx:verify` はアーティファクトと実装の整合確認を行う。XDDPの VERIFY（静的検証）と目的は共通だが、XDDPのVERIFYはCHDのAfterコードと実装コードのファイルレベル乖離チェックまで行う点がより厳格。

## 4. XDDP にあって OpenSpec にないもの

| 機能・特徴 | 説明 | ビジネス価値 |
|-----------|------|------------|
| CR単位ワークスペース管理 | 各CRを独立した `{CR}/` フォルダで統一管理 | 複数並行CRを一覧管理・トレーサビリティ維持 |
| progress.md による15工程ステータス追跡 | ✅/🔄/👀/🔁/⬜ でリアルタイム進捗管理 | 管理職が全体進捗を視認可能 |
| UR/SR/SP 3層要求分類 | ユーザ要求の段階的分解を工程化（Step 02） | 要求漏れ・誤解を早期に発見 |
| SPO（Specout = 母体システム影響範囲調査） | モジュール単位・クロスモジュール単位で分析（Step 04） | 大規模変更の副作用を事前把握 |
| DSN（実装方式の代替案比較） | 複数案の比較と採用理由を Mermaid 図で記録（Step 05） | なぜその方針を選んだのかが設計書に残る |
| 静的検証（VERIFY） | CHDのAfterコード定義と実装コードの乖離を検証（Step 07） | 設計と実装のズレを早期発見 |
| テスト・不具合修正の一体化 | Step 08でテスト仕様→実行→バグ修正→フィードバックを連続処理 | 不具合発見から修正まで文脈を維持 |
| latest-specs/ による最新仕様書自動生成 | モジュール/機能単位で versioned spec を保持（Step 09） | 保守時に「いつの仕様で実装したか」が明確 |
| lessons-learned + improvement-backlog | CRごとの知見・教訓を蓄積・次CRで参照可能 | 組織的学習・品質向上サイクル |
| Excel ↔ Markdown 双方向変換 | CRSをUSDM形式Excelで編集・管理 | 非エンジニア（要件定義者）の参画が容易 |
| マンダトリーレビューループ | 各工程ごとの人間レビュー（最大5回） | 品質保証には寄与するが、開発速度とのトレードオフがある |
| project-steering.md | プロジェクト固有の命名規約・アーキテクチャ決定をAIが参照 | AIが毎回スクラッチで判断する効率低下を解消 |

## 5. 設計思想の本質的な違い

### OpenSpec のアプローチ
**「アーティファクトが存在し合意が取れた状態で実装を始める」**

- フェーズロックなし：任意の順序でアーティファクトを生成し、実装に進める
- ブラウンフィールド優先：既存システムへの変更を主目的とした仕様デルタ方式
- `/opsx:explore` で上流の不確実性を早期解消してから設計へ移行
- グローバル・軽量・30+ AIツール対応
- スタートアップ・プロトタイプ・小〜中規模変更向け

### XDDP のアプローチ
**「人間と AI の協働を前提に、知見を蓄積していく」**

- 各工程でAIが成果物を生成 → 人間がレビュー → AIが修正（最大5ループ）
- UR/SR/SP の3層分解で要求の階層性を明確化
- SPO（母体システム影響調査）で副作用を事前把握
- 知見がログとして蓄積され、次のCRで活用される
- 日本企業・長期保守・大規模変更向け

## 6. XDDPが学べる点（優先度順）

### 優先度: 高

| 項目 | 内容 | 実装状況 | 参考コマンド |
|------|------|---------|------------|
| Explore フェーズの導入 | xddp.02.analysis 前段に `/opsx:explore` 相当の「探索フェーズ」を追加し、要求が曖昧な段階から開始できるようにする | 未実装 | `/opsx:explore` |
| フェーズロック緩和（Quick Mode） | 小規模CRには簡略フロー（xddp.quick）を提供し、OpenSpecのシンプルさとXDDPの堅牢性を融合 | 未実装 | `/opsx:ff` 相当 |

### 優先度: 中

| 項目 | 内容 | 実装状況 | 参考コマンド |
|------|------|---------|------------|
| プロファイルシステム | xddp.config.md に `QUALITY_PROFILE: core/full` を追加し、有効フェーズセットを制御 | 未実装 | core/custom プロファイル |
| lessons-learned のタグスキーマ統一 | `#XDDP_{フェーズ番号}_{フェーズ名}` のように検索可能に統一 | 未実装 | — |
| bulk-close 相当の機能 | 複数CRの一括クローズ・アーカイブ操作 | 未実装 | `/opsx:bulk-archive` |

### 優先度: 低

| 項目 | 内容 | 実装状況 | 参考コマンド |
|------|------|---------|------------|
| 複数AIツール対応の拡大 | xddp.config.md に `AGENT_PROFILE` パラメータを追加し、Cursor/Copilot など複数ツール対応を検討 | 未実装 | `openspec init --tools all` |

## 7. 統合可能性の検討

### 短期（1〜2ヶ月）で実装可能
- **Explore フェーズの薄い追加** — xddp.02.analysis の前段オプションとして `xddp.explore` を追加（要求が曖昧なCR用）
- **lessons-learned タグスキーマの統一** — `#XDDP_{フェーズ番号}_{フェーズ名}` に統一

### 中期（3〜6ヶ月）で検討
- **Quick Mode の設計** — 小規模CR向けの簡略フロー（Step 01 → 03 → 07 → 09 のショートカット）
  - OpenSpec `/opsx:ff` のコンセプトを参考に、XDDP版のファストフォワードパスを定義
- **プロファイルシステムの導入** — xddp.config.md に `QUALITY_PROFILE` パラメータを追加
  - `full`（現行の15工程）、`core`（軽量5工程）など

### OpenSpec × XDDP フェーズマッピング（参考）
| OpenSpec | XDDP |
|----------|------|
| `/opsx:explore` | （相当なし → 追加余地） |
| `/opsx:propose` + specs/ | Step 01〜03（初期化・分析・CRS） |
| `/opsx:verify` | Step 07 VERIFY（静的検証） |
| `/opsx:apply` | Step 07〜08（コーディング・テスト） |
| `/opsx:archive` | Step 09・close（最新仕様・クローズ） |

### 長期（将来）
- 複数AIツール対応（`AGENT_PROFILE` パラメータ）
- `openspec init --tools xddp` 相当のインタラクティブセットアップ

## まとめ

| 特性 | OpenSpec | XDDP |
|------|----------|------|
| 最新バージョン | v1.3.1（2026年4月21日） | — |
| 開発速度 | 最優先（軽量・アクションベース） | 品質と知見蓄積を重視 |
| 自動化 | 高い（AI が提案→設計→タスク生成） | 中程度（人間レビューを必須化） |
| フェーズ制御 | 柔軟（フェーズロックなし・プロファイル切替） | 厳格（15工程・マンダトリー） |
| 対象組織 | グローバル・スタートアップ向け | 日本企業・大規模・継続開発向け |
| レビュー | Optional・軽量 | マンダトリー・厳格 |
| 知見管理 | 仕様に統合（implicit） | 明示的・次CRで活用（explicit） |
| 拡張性 | 30+ AIツール対応・言語中立 | 詳細工程・トレーサビリティ重視 |
| 検証 | `/opsx:verify`（アーティファクト整合確認） | VERIFY（CHD After コードと実装の乖離チェック） |
| ブラウンフィールド | 明示的な設計思想として組み込み | SPO（母体調査）で対応 |

**OpenSpec** は「スピード・軽量・探索的開発」が強み。スタートアップ・プロトタイプ向け。v1.0 以降はフェーズロックを廃止したアクションベースへ移行しており、従来の「3段階プロセス」という説明は旧来の理解。  
**XDDP** は「品質・知見蓄積・大規模変更対応」が強み。日本企業・長期保守向け。

XDDPが OpenSpec から学べる最大のポイントは **「Explore フェーズの導入（要求が曖昧な段階からの開始）」** と **「フェーズロック緩和による Quick Mode」**。なお「仕様デルタ」はXDDPのCRS Before/Afterとして既に実装済みであり、「検証コマンド」はXDDPのVERIFYとして既に実装済み。
