## 🎯 得意な分野（詳細）

### 1. コード波及調査・依存関係分析 ✅ **最高優先度**

**特性**:
- 数千〜数万行のコードを瞬時にスキャン可能
- ファイル間の依存グラフを自動抽出
- リポジトリ間の呼び出し関係を追跡

**研究知見**（arXiv 2026-04-27付）:
- "Static Program Slicing Using Language Models With Dataflow-Aware Pretraining" — 従来の手法より高速で正確

**実運用例**:
- XDDP工程04（specout）で自動化実績
- 従来は人手で3〜5日要していた波及調査が数分で完了

**活用推奨度**: ⭐⭐⭐⭐⭐

---

### 2. 定型的なコード生成

**得意**:
- テストコード（ユニットテスト）
- ボイラープレート（getter/setter、CRUD操作）
- ルーティング定義、設定ファイル

**注意点**:
- 複雑なビジネスロジックは誤りやすい
- 設計書・既存パターンが指針として必須

**活用推奨度**: ⭐⭐⭐⭐

---

### 3. ドキュメント生成・整理

**得意**:
- 要件→仕様→設計書への段階的な文書化
- テンプレートからの一貫性保持
- 既存資料の要約・再構成

**研究知見**:
- Anthropic Constitutional AI — 一貫性のあるドキュメント生成が確認済み

**活用推奨度**: ⭐⭐⭐⭐⭐

## ⚠️ 苦手な分野（詳細）

### 1. 確率的出力（AI幻覚）

**問題**:
- 「もっともらしい」嘘をつく可能性
- 複数回実行で異なる出力

**根本原因**:
- LLMは統計的確率に基づいた次の単語予測
- 事実性保証のメカニズムがない

**対策**（XDDPに組み込み）:
- 各工程でのレビューゲート（AI+人間）
- トレーサビリティにより出力を検証可能に

**活用推奨度**: ❌ 単独で信頼可能なタスクには不適切

---

### 2. 複雑な推論・意思決定

**問題**:
- マルチステップの推論で誤りが蓄積
- 非明示的な制約の評価が困難
- 隠れた仮定の明示化ができない

**研究知見**（arXiv最新）:
- "LLM-as-a-Judge for Human-AI Co-Creation" — AI単独では信頼性が不十分。人間との協力モデル推奨

**具体例**:
- アーキテクチャ決定：複数案の比較は得意 → 最終決定は人間
- トレードオフ評価：複数要因を重み付けする判断は得意 → 優先度決定は人間

**活用推奨度**: ⭐⭐ （人間との組合せが必須）

---

### 3. 確実性が必須な判断

**問題**:
- セキュリティ脆弱性判定が100%正確でない
- コンプライアンス判断で誤りはゼロであるべき

**データベースの危険な操作**:
- DROP TABLE などの破壊的操作の判定
- 手作業による確認が必須

**活用推奨度**: ❌ （必ず人間による最終確認）

---

### 4. 新規ドメイン知識の創造

**問題**:
- 学習データにない領域では信頼性が著しく低下
- 業界固有の慣例や隠れた制約を知らない

**例**:
- 新規技術スタックの導入判断
- 業界固有の規制対応

**活用推奨度**: ⭐ （専門家による指導下でのみ）

## 🔄 XDDPプロセスが提供する「AIの苦手を補う仕組み」

### レビューゲートの役割

```
工程②（AI分析）
  ↓
AIレビュー（矛盾チェック）← 同じAIが自己検証
  ↓
人間レビュー（常識判定） ← 「それって本当？」という確認
  ↓
工程③（確定）
```

### トレーサビリティの役割

```
要求 → 仕様 → 設計 → コード
  ↑     ↑     ↑      ↑
  └─────┴─────┴──────┘
      （追跡可能性）

「このコード行は、どの要求から出たのか」を遡れるため、
AI出力の正当性を事後的に検証可能
```
