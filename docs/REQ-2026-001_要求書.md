## 1. 背景と目的

### 1.1 背景

XDDPは対応漏れを防ぎ品質を担保するために体系化された派生開発プロセスであり、効率的なソフトウェア開発手法として有効である。しかし、プロセスの習得には学習コストが高く、各工程で求められる成果物の書き方や観点をベテラン開発者と同じレベルで実践できるようになるまでに時間を要する。この結果、開発経験が浅い開発者が担当した場合に成果物の品質にばらつきが生じ、対応漏れや手戻りのリスクが高まる。

### 1.2 目的

生成AIがXDDP各工程の成果物作成・レビューを担うことで、以下を実現する。

- **工数削減**：成果物のドラフト生成・レビュー・修正をAIが自動化し、人の作業を確認・判断に集中させる
- **品質の均質化**：開発経験に依らず、ベテランと同水準の観点を網羅した成果物（対応漏れなし）を生成する
- **要求分析品質の向上**：AIが要求を多角的に分析・整理することで、曖昧な要求の明確化や見落としを早期に検出し、上流工程での品質を高める
- **知見・ノウハウの蓄積**：仕様書が存在しない状態でもスペックアウトによりコードから仕様を逆引きして文書化することで、暗黙知の形式知化・仕様の明確化・開発ノウハウの蓄積を継続的に行う
- **プロセス定着**：AIがXDDPの手順をガイドすることで、経験の浅い開発者のXDDP習得を支援する

### 1.3 適用範囲

本システムはClaude Codeのスキル・エージェント群として実装し、ソフトウェア開発プロジェクト上で使用するCLIベースのツール群として提供する。

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| XDDP | eXtreme Derivative Development Process。派生開発に特化した開発プロセス |
| CR | Change Request。変更要求の単位 |
| スペックアウト | 母体コードを調査し既存仕様を文書化する作業 |
| TM | Traceability Matrix。要求〜仕様〜設計〜実装の対応を管理する表 |
| 変更要求仕様書 | ユーザ要求・システム要求・仕様の3階層で変更内容を定義した文書 |
| 変更設計書 | 変更仕様を実装レベルで記述した設計文書 |
| 母体 | 変更前の既存ソースコード・仕様書群 |
| 既存の設計書 | CR適用前の時点でプロジェクトルートの `latest-specs/` に存在するシステム設計文書。スペックアウト時の参照元となる（初回CR時は存在しない場合がある） |
| 最新仕様書 | 既存の設計書・スペックアウト資料・変更設計書を統合した変更適用後のシステム仕様。`latest-specs/` に上書き保存され、次のCRの「既存の設計書」となる |

## 3. ユーザ要求

### 3.1 工程とAI担当範囲

| 4 | スペックアウト（母体調査） | **AI** | スペックアウト資料 |

### 3.2 AIレビュー要求

- **UR-001**：各工程でAIが成果物を生成した後、その工程に適した知識を持つレビュアーエージェントが成果物をレビューすること
- **UR-002**：レビューで指摘事項が発生した場合、AIが自動的に修正を行うこと
- **UR-003**：修正後、再レビューを実施し、重大・中程度の指摘がなくなるまでループすること。ループ上限は5回とし、上限到達時は「未解決の重大指摘あり」として記録し、人間の判断に委ねること
- **UR-015**：レビューは成果物を生成したエージェントとは別コンテキスト（独立したエージェント）で実行すること（生成時のコンテキストに引きずられない客観的なレビューを担保するため）

### 3.3 コマンドインタフェース要求

- **UR-004**：各工程をClaude Codeのスキルコマンドとして呼び出せること
- **UR-005**：工程を順番に実行するオーケストレーターコマンドを持つこと
- **UR-006**：特定工程から再開できること（前工程の成果物を入力として利用）
- **UR-007**：CR番号で成果物を管理できること
- **UR-017**：人が編集したExcel形式の変更要求仕様書をMarkdownに変換するコマンドを提供すること
- **UR-036**：コマンド名はXDDPコマンドであることが明示でき、かつ実行順序がわかる命名規則に従うこと。具体的には以下の形式とする

  ```
  xddp.{NN}.{phase}            # 各工程のメインコマンド（NNは2桁の工程番号）
    xddp.{NN}.{phase}.review   # 同工程のAIレビューコマンド
    xddp.{NN}.fix.{phase}.review      # 同工程のレビュー指摘に基づく修正コマンド
  ```

  例：
  ```
  xddp.01.req                  # 変更要求仕様書作成
    xddp.01.req.review         # 変更要求仕様書レビュー
    xddp.01.req.fix            # 変更要求仕様書修正
  ```

  - レビューコマンドを持たない工程は `xddp.{NN}.{phase}` のみとする
  - `xddp.status` など工程に紐づかないユーティリティコマンドは番号なしとする

### 3.4 進捗管理要求

- **UR-011**：CR単位で現在どの工程まで完了しているかを確認できること
- **UR-012**：各工程の状態（未着手・進行中・完了・レビュー待ち・差し戻し）を一覧で表示できること
- **UR-013**：進捗確認は専用コマンド一つで実行できること
- **UR-014**：工程完了時に次工程への案内（次に実行すべきコマンド）を表示すること

### 3.5 成果物管理要求

- **UR-008**：全成果物はMarkdown形式でファイルとして保存すること
- **UR-009**：成果物はCR番号をルートとするディレクトリに格納し、工程順の番号を付けた英語フォルダ名で管理すること（下記フォルダ構成参照）
- **UR-010**：各成果物はバージョン管理（版数・変更履歴）を持つこと
- **UR-016**：変更要求仕様書はMarkdownに加えてExcel形式でも出力できること

#### 成果物フォルダ構成

```
{プロジェクトルート}/
├── latest-specs/           # 最新仕様書（プロジェクト共通）
│   └── ...                   #   CR開始時は参照元、工程15完了時に更新される
└── {CR番号}/
    ├── 01_requirements/      # 要求書
    ├── 02_analysis/          # 要求分析メモ
    ├── 03_change-requirements/  # 変更要求仕様書（Excel含む）
    ├── 04_specout/           # スペックアウト資料
    ├── 05_architecture/      # 方式検討メモ
    ├── 06_design/            # 変更設計書
    ├── 07_coding/            # コーディング差分・メモ
    ├── 08_code-review/       # 静的検証レポート
    ├── 09_test-spec/         # テスト仕様書
    ├── 10_test-results/      # テスト結果記録
    ├── review/               # 各工程のレビュー結果
    │   ├── 02_analysis-review.md
    │   ├── 03_change-requirements-review.md
    │   ├── 05_architecture-review.md
    │   ├── 06_design-review.md
    │   ├── 09_test-spec-review.md
    │   └── ...
    └── progress.md           # 進捗管理
```

- **UR-027**：AIが成果物を生成した各工程のレビュー結果は `review/` フォルダに格納すること
- **UR-028**：レビュー結果ファイルには指摘事項と対応内容を対で記録し、対応状況（未対応・対応済・対応不要）を明示すること
- **UR-029**：既存の設計書が存在する場合、プロジェクトルート直下の `latest-specs/` フォルダに格納すること。スペックアウト（工程4）の参照元として使用すること。既存仕様書の形式はプロジェクトに依存するため本システムでは規定せず、AIが形式を解釈して処理すること
- **UR-030**：最新仕様書（工程15）は、`latest-specs/`（存在する場合）・`04_specout/`・`06_design/` を統合し、プロジェクトルートの `latest-specs/` を直接更新すること。CR完了後の `latest-specs/` が次のCRの「既存の設計書」となる
。
- **UR-035**：CRの粒度は1機能=1CRを基本とし、規模はプロジェクトに依存するため本システムでは規定しない。ただし以下を参考目安として提示し、超過時は分割を推奨すること
  - スペックアウトの波及先ファイル数：20ファイル超で分割を検討
  - 変更コード行数：500行超で分割を検討
  - 1CR内に独立した関心事（例：DB変更・UI変更・API変更）が混在する場合は分割を検討
  - テスト確認項目数：50項目超で分割を検討

### 3.6 テスト要求

#### 単体テスト

- **UR-021**：対象コードの実装言語に応じた適切なテストフレームワークを自動選択して適用すること
- **UR-022**：単体テストはC0（命令網羅）およびC1（分岐網羅）カバレッジ100%を達成すること
- **UR-023**：カバレッジ計測はフレームワーク標準のツールを使用し、結果をテスト結果記録に記載すること
- **UR-024**：カバレッジ100%未達の場合はテストNGとして扱い、不具合修正フローに移行すること

#### 結合テスト

- **UR-025**：結合テストの方針・範囲・環境はプロジェクトごとに要求書で定義すること（実装対象システムに依存するため）
- **UR-026**：要求書にテスト要求セクションを設け、結合テストの種別・対象・合否基準を記述できること
- **UR-031**：AIが担う結合テストの自動化範囲はテスト仕様書の作成までとすること。テスト実行の自動化可否はプロジェクト環境に依存するため、実行・判定は人間が行うこと

### 3.6.5 変更要求仕様書Excelフォーマット要求

- **UR-037**：変更要求仕様書のExcel形式はUSDMの表構造（カテゴリ名・記号、要求、要求ID、理由、説明、仕様グループ名、仕様ID）に準拠すること（AFFORDD T2研究会 USDM小冊子 基礎編 参照）
- **UR-038**：仕様の変更前・変更後は、仕様本文セルとは独立したセルに記述すること。Excelでは仕様行の直後に before/after 行を設け、以下の形式とする

  | セル | 内容 |
  |------|------|
  | `before` | 変更前の仕様（新規追加の場合は「なし」） |
  | `after` | 変更後の仕様 |

- **UR-039**：変更前・変更後セルにより、変更の差分が仕様単位で追跡可能であること
- **UR-040**：要求・仕様の各行に更新日（YYYY-MM-DD）および更新者（人名またはAIエージェント名）を記録するセルを設けること。変更のたびに当該セルを更新し、要求・仕様の変遷が追跡できること

---

### 3.7 人・AIハンドオフ要求

> AI生成の成果物を人が確認・編集し、AIプロセスに戻すための要求

- **UR-018**：変更要求仕様書はExcel上で人がレビュー・編集できる形式で出力すること
- **UR-019**：人によるExcel編集後、その内容をMarkdownに変換してAIの後続工程への入力として使用できること
- **UR-020**：Excel→Markdown変換時、人の編集内容（追記・修正・削除）が正確に反映されること

### 3.8 人承認フロー要求

- **UR-033**：AIによる各工程完了後、人がレビューを実施できること。指摘事項はAIへの修正依頼または人による直接修正のいずれかで対応できること
- **UR-034**：将来フェーズとして、人の承認対象を最終成果物（変更後ソースコードおよびテスト結果記録）のみに絞り込めるモードを提供すること。それ以外の工程はAIが自律的に完結し、人の介入を不要とすること

## 4. システムへの制約条件

- Claude Code（CLI）上のスキル・エージェントとして実装すること
- 外部サービスへの依存を最小化すること
- 既存のプロジェクトに対して後付けで適用可能なこと
- 日本語での成果物作成に対応すること
- スペックアウト対象のソースコード・既存ドキュメントは英語・日本語・混在のいずれであっても処理できること（UR-032）
- エージェントへの指示・スキルプロンプト・エージェント間メッセージは英語で記述すること（トークン消費の削減およびモデルの推論精度向上のため）
- 成果物（仕様書・設計書・テスト文書等）の出力内容およびユーザへの応答メッセージは日本語とすること
