# GitHub Spec kit vs XDDP 比較分析

作成日: 2026-04-22  
最終更新: 2026-04-26

## 概要

| ツール | 提供元 | 設計思想 |
|--------|--------|---------|
| GitHub Spec kit | GitHub（オープンソース） | 仕様駆動開発（Spec-Driven Development）、マルチAIエージェント対応 |
| XDDP | 本リポジトリ | eXtreme Design-Driven Process、日本企業向け詳細工程管理 |

### 参考リンク
- [GitHub Spec Kit Repository](https://github.com/github/spec-kit)
- [GitHub Blog - Spec-driven development with AI](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/)

---

## 1. GitHub Spec kit とは

GitHubが公開したオープンソースの仕様駆動開発ツールキット。「仕様を実行可能な成果物として扱う」**Intent-Driven Development** が核心思想。

### 対応 AI エージェント
GitHub Copilot、Claude Code、Gemini CLI など **30以上** のAIコーディングエージェントに対応。

### 3つの開発パターン

| パターン | 内容 |
|---------|------|
| **グリーンフィールド（0→1）** | ゼロから新規構築するプロジェクト |
| **創造的探索** | 複数の技術スタックを並列検証する実験的開発 |
| **ブラウンフィールド（反復拡張）** | 既存システムへの機能追加・改修 |

### 5つのコアコマンド

| コマンド | 内容 |
|---------|------|
| `/speckit.constitution` | プロジェクト原則・制約の確立 |
| `/speckit.specify` | ユーザージャーニー・成功指標を言語化（技術的詳細は後） |
| `/speckit.plan` | 技術スタック・アーキテクチャ・制約・コンプライアンス要件の決定 |
| `/speckit.tasks` | 仕様を小さなレビュー可能な単位に分解 |
| `/speckit.implement` | タスク実行（AI が既に明確な指示を持つ） |

### 主なコンポーネント
- **specify-cli** — `uv tool install specify-cli` でインストール可能なプロジェクト初期化・テンプレート管理CLI
- **構造化テンプレート** — Constitution（原則）/ Specification（仕様）/ Technical Plan（実装計画）
- **`.kiro/steering/`** — プロジェクトメモリ管理（命名規約・アーキテクチャ決定・既存パターン）
- **gap-analysis フレームワーク** — 「既存資産 vs 新要求」を Option A/B/C の3択で比較
- **コミュニティ拡張** — Jira統合・Azure DevOps連携・セキュリティレビューなど **80以上** の拡張が利用可能

## 2. プロセス体系の比較

| 観点 | GitHub Spec kit | XDDP |
|------|-----------------|------|
| フェーズ数 | 5コマンド体系（シンプル・グローバル向け） | 15工程（詳細・日本企業向け） |
| 開発対象スコープ | 単一機能（per feature）、新規・既存・探索の3パターン | 単一変更要求（CR）・大規模変更対応 |
| マルチリポジトリ対応 | 対応（プリセット・拡張で対応） | 対応（xddp.config.md の MULTI_REPO フラグ、2026-04-22〜） |
| 自動化度 | 高い（タスク単位の自動実行） | 中程度（AI→人間→修正の反復） |
| 人間の介入 | ゲート（承認点）のみ | 各工程のレビュー＋修正ループ（最大5回） |
| 知見管理 | コミュニティ拡張で対応（標準機能なし） | lessons-learned.md + improvement-backlog.md |
| トレーサビリティ | 軽量（タスクフロー中心） | 詳細（UR→SR→SP の段階的分解・CHDで一対一対応） |
| 文書形式 | Markdown推奨（柔軟） | Markdown + Excel（USDM形式）・標準化厳密 |
| 国際化 | 30以上のAIツール対応・英語中心 | 日本語ベース・特定プロセス向け |
| テスト位置づけ | 実装ワークフロー内に埋め込み | 独立した工程（Step 08：テスト設計〜不具合修正） |
| 品質ゲート外部化 | 拡張機能で対応 | xddp.06.rules.md / xddp.07.rules.md で外部化（2026-04-26〜） |

## 3. GitHub Spec kit にあって XDDP にないもの

| 機能・特徴 | 説明 | ビジネス価値 |
|-----------|------|------------|
| マルチプラットフォーム統一 | Cursor, Copilot, Windsurf, Gemini など **30以上** のAIツールを同一ワークフローで統一 | 組織が使用するAIツールの選択肢が自由 |
| コミュニティ拡張（80以上） | Jira統合・Azure DevOps連携・セキュリティレビューなど多様なプラグインが利用可能 | 既存ツールチェーンとの連携を追加コードなしで実現 |
| `.kiro/steering/` プロジェクトメモリ管理 | 命名規約・アーキテクチャ決定・既存パターンをAIが反復参照できる一元管理 | AIが毎回スクラッチから判断する効率低下を解消 |
| gap-analysis フレームワーク | 「既存資産 vs 新要求」を Option A/B/C の3択で体系的に比較 | 実装方式選択の意思決定が可視化 |
| 速度別パス選択 | Quick / Step-by-step / Batch の3パスから選択可能 | 案件の緊急度に応じた柔軟対応 |
| 3つの開発パターン対応 | グリーンフィールド（新規）・創造的探索（並列検証）・ブラウンフィールド（既存改修）を明示的に区別 | プロジェクトタイプに応じた適切なアプローチを選択可能 |

> **注:** `.kiro/steering/` 相当の `project-steering.md` は2026-04-22のアップデートでXDDPに導入済み。

## 4. XDDP にあって GitHub Spec kit にないもの

| 機能・特徴 | 説明 | ビジネス価値 |
|-----------|------|------------|
| CR単位ワークスペース管理 | 各CRの全15工程を `{CR}/` フォルダで統一管理 | 複数の並行CRを一覧管理・トレーサビリティ維持 |
| progress.md による15工程ステータス追跡 | ✅/🔄/👀/🔁/⬜ でリアルタイム進捗管理 | 管理職が全体進捗を視認可能 |
| UR/SR/SP 3層要求分類 | ユーザ要求の段階的分解を工程化（Step 02） | 要求漏れ・誤解を早期に発見 |
| SPO（Specout = 母体システム影響範囲調査） | モジュール単位・クロスモジュール単位で完全分析（打ち切りなし）（Step 04） | 大規模変更の副作用を事前に網羅把握 |
| DSN（実装方式の代替案比較） | 複数案の比較と採用理由を Mermaid 図で記録（Step 05） | なぜその方針を選んだのかが設計書に残る |
| コーディング品質ゲートの外部化 | `xddp.07.rules.md` にエラーハンドリング・セキュリティ・ログ・CHD適合性ルールを定義（Step 07） | ルールをコードに触れずにカスタマイズ可能 |
| 設計品質ゲートの外部化 | `xddp.06.rules.md` に設計ルール（フェイルセーフ・ロールバック設計含む）を定義（Step 06） | 設計品質の組織標準をスキルと分離して管理 |
| 静的検証（VERIFY） | CHDのAfterコード定義と実装コードの乖離を検証（Step 07） | 設計と実装のズレを早期発見 |
| テスト・不具合修正の一体化 | Step 08でテスト仕様→実行→バグ修正→フィードバックを連続処理 | 不具合発見から修正まで文脈を維持 |
| latest-specs/ による最新仕様書自動生成 | モジュール/機能単位で versioned spec を保持（Step 09） | 保守時に「いつの仕様で実装したか」が明確 |
| lessons-learned + improvement-backlog | CRごとの知見・教訓を蓄積・次CRで参照可能 | 組織的学習・品質向上サイクル |
| マルチリポジトリ対応 | `xddp.config.md` の `MULTI_REPO/REPOS` 設定でリポジトリ境界をまたいだ調査・実装が可能 | モノリポ以外のマルチリポ構成でXDDPを適用可能 |
| Excel ↔ Markdown 双方向変換 | CRSをUSDM形式Excelで編集・管理 | 非エンジニア（要件定義者）の参画が容易 |

## 5. 設計思想の本質的な違い

### GitHub Spec kit のアプローチ
**「仕様の質を極めることで、実装を機械化する」**

- 前段階（Constitution → Specify → Plan → Tasks）で完璧な仕様を作る
- Implement 段階では AI が「ほぼ機械的に」実装（自動化度が高い）
- 人間はゲート（承認点）でのみ介入
- 30以上のAIツールと80以上のコミュニティ拡張によるエコシステム志向

### XDDP のアプローチ
**「人間と AI の協働を前提に、知見を蓄積していく」**

- 各工程で AI が成果物を生成 → 人間がレビュー → AI が修正（ループ最適化済み）
- UR/SR/SP の3層分解で要求の階層性を明確化
- 品質ゲートを外部ファイル（*.rules.md）として分離・カスタマイズ可能
- 知見がログとして蓄積され、次のCRで活用される

## 7. 統合可能性の検討

### 短期（1〜2ヶ月）で実装可能
- lessons-learned タグスキーマの統一化（`#XDDP_{フェーズ番号}_{フェーズ名}`）
- gap-analysis テンプレートの ANA への導入

### 中期（3〜6ヶ月）で検討
- GitHub Spec kit と XDDP の「フェーズマッピング」ドキュメント作成
  - Constitution → xddp.config.md + project-steering.md
  - Specify → Step 01〜03
  - Plan → Step 04〜05
  - Tasks → Step 06
  - Implement → Step 07〜09
- rules.md 形式で外部共有可能な XDDP 拡張キットの整備（コミュニティ拡張相当）

### 長期（将来）
- XDDP が複数言語対応する場合、GitHub Spec kit の多言語テンプレート管理を参考に導入
- 30以上のAIツール対応に向けた XDDP プロセス定義の汎用化（Claude Code 固有の記法を抽象化）

## まとめ

**GitHub Spec kit** は「グローバル対応・高自動化・30以上のAIツール統一・80以上のコミュニティ拡張」が強み。  
**XDDP** は「人間協働・日本企業規模対応・知見蓄積・強固なトレーサビリティ・品質ゲートの外部化」が強み。

両者は補完的な関係にあり、2026-04-22〜2026-04-26のアップデートでXDDPが学ぶべき主要な要素（project-steering.md・フェーズ別品質ゲート外部化・マルチリポジトリ対応）はほぼ導入済み。次の優先項目は **lessons-learned のタグスキーマ統一** と **gap-analysis フレームワークの導入**。
