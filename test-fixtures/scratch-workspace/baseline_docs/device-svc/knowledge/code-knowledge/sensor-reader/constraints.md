## 既知の制約・落とし穴

### [CK-001] 閾値判定の境界値（同値）が漏れていた off-by-one
**対象:** `src/sensor_reader.py` の `read_sensor()`（および `THRESHOLD_HIGH` を用いた閾値比較全般）
**内容:** 閾値比較に `>` を用いており、センサー値が `THRESHOLD_HIGH` と同値の場合にアラートが発火しない不具合があった（TC-101 / NG-001）。`>` を `>=` に修正済み。同様のパターンが他の閾値判定にも潜んでいる可能性があるため、`sensor_reader.py` 内の他の閾値判定箇所も棚卸しが必要。
**回避策:** 閾値判定を追加・変更する際は `>` と `>=` のどちらが要求に合致するかを必ず確認する。境界値（同値）のテストケースを必ず用意する。
**出典:** CR-2026-900 / NG-001 / 発見日: 2026-06-21

## パフォーマンス・非機能特性

### [CK-002] 読み取りループのポーリング間隔固定
**観察:** `src/sensor_reader.py:read_sensor()` の読み取りループはポーリング方式で1秒間隔固定になっている。
**示唆:** ラベル付加によりレスポンスサイズが増えるため、ラベル関連の変更を行う際は間隔調整の必要性を検討すること。
**出典:** CR-2026-900（SPO Section 5.6 より）

### [CK-003] 閾値比較が境界値を含まない実装（後方互換上の注意）
**観察:** `src/sensor_reader.py:THRESHOLD_HIGH` の閾値比較が `>` のみで境界値を含まない実装になっている（CK-001 で修正済みだが、設計判断として記録）。
**示唆:** ラベル別閾値を将来追加する場合は、境界条件（同値の扱い）を仕様化する必要がある。
**出典:** CR-2026-900（SPO Section 5.6 より）
