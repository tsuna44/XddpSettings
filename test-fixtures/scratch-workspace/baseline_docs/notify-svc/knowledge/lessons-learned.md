# lessons-learned（notify-svc・クローズ済み）

### LL-002：フィルタ条件「空配列=全件対象」は明示的にテストすべき

**CR：** CR-2026-900 ／ **工程：** コーディング ／ **repo：** notify-svc ／ **タグ：** #コーディング

**発生状況：**
`NotificationTarget.label_filter` が空配列の場合に「全件対象」として扱う仕様を実装する際、空配列の意味（フィルタなし）を実装者が見落としやすいことに気づいた。

**学んだこと：**
「空＝制限なし」という暗黙ルールを持つフィルタ条件は、コード上のコメントとテストケースの両方で明示しないと意図が伝わらない。

**次回への適用：**
- 空配列・null・未設定のいずれかに特別な意味を持たせるフィールドは、設計書（CHD）に明記する
- 対応するテストケースを必ず1件以上用意する

### LL-{破壊的変更警告}：device.alert.raised の破壊的変更

**CR：** CR-2026-900 ／ **工程：** 設計 ／ **repo：** notify-svc ／ **タグ：** #設計

**発生状況：**
破壊的インタフェース変更あり（CR: CR-2026-900）。device.alert.raised の旧バージョン（labels フィールドなし）への依存コードを確認すること。

**学んだこと：**
イベントペイロードへの必須フィールド追加は breaking 変更として扱い、提供側・消費側の同時デプロイが前提になる。

**次回への適用：**
- notify-svc.notifier が labels フィールド欠損時にどう振る舞うか確認済みであることを再確認する
